## 一、项目概况

本项目用于实现 Markdown 格式文档的数据清理，通过自动化流程对输入目录中的文档进行去噪、编码统一、表格处理、错误修复、去重、格式规范化、LLM 摘要生成与逻辑关联分析等处理，最终输出结构清晰、格式标准化的 Markdown 文件。

项目支持多种输入文件格式（Markdown、Excel、PDF、Word、PPT、图片等），通过 MinerU API 或本地转换器将非 Markdown 文件自动转换为 Markdown 格式后处理。

## 二、功能需求

### 2.1 数据清理功能

| 编号 | 功能名称 | 功能说明 | 验收标准 |
|------|----------|----------|----------|
| F1 | 去噪处理 | 移除文档中的无关内容，包括：页码（`Page N`、`- N -`、`[N]`、`N/M` 等格式）、连续空行（3行以上缩减为2行）、多余空格（3个以上连续空格缩减为1个）、制表符（替换为4个空格）、行尾空白字符 | 页码格式全部移除；连续空行不超过2行；无3个以上连续空格；代码块内制表符和空格保持原样 |
| F2 | 统一编码和符号 | 统一使用 UTF-8 编码；根据上下文语境统一全半角标点（中文语境下半角→全角，英文语境下全角→半角）；清理不可见字符（零宽空格、零宽连接符、BOM等）；统一换行符为 `\n`；标准化 OCR 误识别符号（如 `≡`→`=`、全角空格→半角空格） | 输出文件为有效 UTF-8 编码；代码块和内联代码内的标点不被修改；不可见字符全部移除 |
| F3 | 表格处理 | 将 HTML `<table>` 元素转换为标准 Markdown 表格；合并因换行断裂的表格行；标准化表头分隔行（确保使用 `---`）；表格行识别约束（必须以 `\|` 开头或结尾且包含至少2个 `\|`，避免误判普通文本） | HTML 表格正确转换为 Markdown 表格；断裂表格行正确合并；普通文本不被误判为表格 |
| F4 | 重复数据消除 | 使用 MD5 哈希对段落精确去重，保留首次出现的段落；代码块提取为占位符保护，避免代码块内重复内容被误删；用双换行重新组合唯一段落 | 完全相同的段落仅保留首次出现；代码块内重复内容不受影响 |
| F5 | 格式规范 | 标题格式：`#` 后确保一个空格，剥离尾部闭合 `#`；列表格式：统一无序列表标记，标准化有序列表格式；代码块前后确保有空行；水平线统一为 `---` 并确保前后有空行；移除行尾空白；标题前后确保有空行；表格格式标准化 | 输出文件符合标准 Markdown 格式规范，可被常见 Markdown 渲染器正确解析 |
| F6 | 修复错误 | 修复被误识别为 Markdown 标题的代码注释（如 `# systemctl start httpd`，含中文的标题不转义）；修复跨行断裂的标题（如 `# 三、` + `防火` + `# 墙`）；合并段落内的断裂行和多余换行（PDF 转换常见问题）；修复行首多余空格（保留列表项缩进最多3空格）；确保代码块前后有空行 | 代码注释不被误渲染为标题；断裂标题正确合并；段落内无多余换行；列表缩进保留 |

### 2.2 文档摘要与逻辑关联

| 编号 | 功能名称 | 功能说明 | 验收标准 |
|------|----------|----------|----------|
| F7 | 文档摘要 | 使用 LLM 阅读完整文件内容，提炼 200 字符以内的文档概要说明；使用 LLM 为每个大段落生成 50 字以内的章节摘要；在每个大段落标题下方添加文档概要和本段概要，以 `---` 分隔的上下文块形式嵌入 | 文档概要不超过 200 字符；章节摘要不超过 50 字符；上下文块格式为 `---\n文档概要：...\n本段概要：...\n逻辑关联：...\n---`（逻辑关联字段可选，无关联信息时省略） |
| F8 | 添加逻辑关联 | 在每个大段落标题下方添加关联关系说明，包含两类：**结构关系**（所属章节、包含小节、上一节、下一节）和 **语义关系**（通过 LLM 分析的前置依赖、后续应用、因果关系等）；合并为"逻辑关联"字段嵌入上下文块 | 逻辑关联字段包含结构关系和语义关系两部分，以分号分隔 |
| F9 | 大段落判定 | 取文档中实际出现的最高层级标题和次高层级标题作为大段落；子级段落（如三级以下标题）不添加上下文信息 | 仅一级和二级标题（或文档最高两层标题）下方嵌入上下文块 |
| F10 | 目录识别与跳过 | 自动识别文档目录部分（匹配 `数字 + 标题 + ... + 页码` 格式），不对目录条目添加上下文信息 | 目录部分不嵌入任何上下文块 |
| F11 | 旧上下文清理 | 处理前自动清理已有的旧上下文块和旧摘要信息，确保重复运行结果幂等 | 重复运行输出一致，无残留旧摘要 |

### 2.3 文件格式转换

| 编号 | 功能名称 | 功能说明 | 验收标准 |
|------|----------|----------|----------|
| F12 | MinerU API 精准解析 | 调用 MinerU 精准解析 API 将 PDF、Word、PPT、图片等文件转换为 Markdown；支持批量上传（≤200文件）；文件大小 ≤ 200MB，页数 ≤ 600 页；返回 ZIP（Markdown + JSON） | 精准模式成功返回 Markdown 文本 |
| F13 | MinerU API Agent 轻量解析 | 调用 MinerU Agent 轻量解析 API 进行快速转换；一次只能解析一个文件（强制 batch_size=1）；文件大小 ≤ 10MB，PDF 页数 ≤ 20 页；返回 Markdown CDN 链接 | Agent 模式成功返回 Markdown 文本；超限文件自动跳过 |
| F14 | 本地转换回退 | MinerU API 所有模式均失败后，自动回退到本地转换；本地转换支持 DOCX（python-docx）、PDF（pdfplumber）、Excel（openpyxl）格式；需在配置文件中启用 `fallback_local_convert: true` | MinerU 失败后自动尝试本地转换；不支持的格式返回 None 并记录日志 |
| F15 | Excel 直接转换 | Excel 文件（.xlsx/.xls）使用 pandas 直接转换为 Markdown 表格，不经过 MinerU API 和 LLM 处理；每个工作表转为独立 Markdown 表格，以 `## 工作表名称` 作为标题；转换后使用基础清洗链（不含 LLM） | Excel 文件正确转为 Markdown 表格；输出文件扩展名为 `.md`；不调用 MinerU API 和 LLM |

## 三、文件处理

### 3.1 输入与输出

- **输入目录**：通过命令行参数指定，包含待处理的文件
- **输出目录**：通过命令行参数指定，存放清理后的文件（不存在则自动创建）
- **命令行格式**：`python main.py <输入目录> <输出目录> [--config 配置文件] [--mode precision|agent]`

### 3.2 支持的文件类型

| 文件类型 | 扩展名 | 处理方式 |
|----------|--------|----------|
| Markdown | `.md`、`.markdown` | 直接清洗（完整清洗链） |
| Excel | `.xlsx`、`.xls` | pandas 直接转换为 Markdown 表格，基础清洗链（不含 LLM） |
| PDF | `.pdf` | MinerU API 转换后清洗，或本地 pdfplumber 转换 |
| Word | `.doc`、`.docx` | MinerU API 转换后清洗，或本地 python-docx 转换 |
| PPT | `.ppt`、`.pptx` | MinerU API 转换后清洗 |
| 图片 | `.png`、`.jpg`、`.jpeg`、`.bmp`、`.tiff`、`.tif` | MinerU API 转换后清洗 |
| 网页 | `.html`、`.htm` | MinerU API 转换后清洗 |
| 其他 | `.xps`、`.epub` | MinerU API 转换后清洗 |

### 3.3 MinerU 转换回退策略

采用三级回退策略确保转换成功率：

1. **第一级**：使用指定模式（precision 或 agent）调用 MinerU API
2. **第二级**：若 precision 模式失败，自动尝试 Agent 轻量模式
3. **第三级**：若所有 MinerU 模式均失败，回退到本地转换（需启用 `fallback_local_convert`）

### 3.4 批量处理

- 输入目录中的文件按类型分类后分批处理
- 每个批次处理的文件数量在配置文件中配置，**默认 2 个文件/批次**
- Agent 轻量解析模式强制 batch_size=1（一次只能解析一个文件）
- 临时目录管理：MinerU 转换后的中间文件存放在临时目录，处理完成后自动清理

## 五、配置要求

配置文件为项目根目录下的 `config.yaml`，包含以下三部分：

### 5.1 LLM 配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `llm.api_key` | LLM API 密钥 | 无（必填） |
| `llm.base_url` | OpenAI 兼容 API 地址 | `https://api.openai.com/v1` |
| `llm.model` | 模型名称 | `gpt-4o-mini` |
| `llm.timeout` | 请求超时时间（秒） | 120 |
| `llm.temperature` | 生成温度 | 0.3 |
| `llm.max_tokens` | 最大生成 token 数 | 8192 |

### 5.2 处理配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `processing.batch_size` | 每批次处理的文件数量 | 2 |

### 5.3 MinerU 配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `mineru.api_key` | MinerU API 密钥（精准解析模式必填） | 无 |
| `mineru.mode` | 解析模式：`precision`（精准）或 `agent`（轻量） | `precision` |
| `mineru.model_version` | 模型版本：`pipeline`、`vlm`（推荐）或 `MinerU-HTML` | `vlm` |
| `mineru.base_url` | MinerU API 基础地址 | `https://mineru.net` |
| `mineru.batch_upload_url` | 批量上传 API 路径（仅精准解析模式使用） | `https://mineru.net/api/v4/file-urls/batch` |
| `mineru.poll_interval` | 轮询间隔（秒） | 15 |
| `mineru.poll_timeout` | 轮询超时（秒） | 300 |
| `mineru.fallback_local_convert` | MinerU 转换失败后是否尝试本地转换 | `true` |

## 六、日志与监控

- 日志文件存放于项目根目录下的 `log/` 目录，按时间戳命名（格式：`clean_YYYYMMDD_HHMMSS.log`）
- 执行过程中同时输出到控制台和日志文件

- 日志文件输出级别：DEBUG 及以上

- 每个处理器的执行耗时记录到日志

## 七、清洗器处理器链

清洗器按固定顺序执行处理器链，每个处理器接收上一个处理器的输出作为输入：

### 7.1 完整清洗链（适用于 Markdown、PDF、Word 等文档型文件）

| 顺序 | 处理器 | 职责 |
|------|--------|------|
| 1 | EncodingProcessor | 编码统一、不可见字符清理、全半角标点转换 |
| 2 | DenoiseProcessor | 去噪（页码、空行、空格、制表符） |
| 3 | TableProcessor | 表格处理（HTML 转 Markdown、行合并） |
| 4 | ErrorProcessor | 错误修复（断裂标题、段落断裂、行首空格） |
| 5 | DedupProcessor | 去重（MD5 精确去重） |
| 6 | FormatStandardizationProcessor | 格式标准化（标题、列表、代码块、水平线） |
| 7 | SegmentIntroProcessor | LLM 摘要与逻辑关联（文档概要、章节摘要、逻辑关系） |

### 7.2 基础清洗链（适用于 Excel 等数据型文件）

仅包含前 6 个处理器，不含 SegmentIntroProcessor（LLM 处理）。

## 八、错误处理

| 场景 | 处理方式 |
|------|----------|
| 输入目录不存在 | 报错退出（exit code 1） |
| MinerU 模式参数非法 | 报错退出（exit code 1） |
| 单个文件处理失败 | 跳过该文件，记录错误日志，继续处理下一个 |
| MinerU API 转换失败 | 按回退策略尝试下一级转换 |
| LLM API 调用失败 | 文档概要/章节摘要/逻辑关联留空，不影响其他处理 |
| 文件编码无法识别 | 按优先级尝试 UTF-8 → GBK → GB2312 → GB18030，均失败则报错 |
